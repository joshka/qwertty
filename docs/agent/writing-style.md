# Writing Style

Project prose should be concise, direct, and accurate.

## Rules

- State current behavior, not aspirations.
- Put the useful answer before background.
- Prefer examples before theory for user docs.
- Keep maintainer context out of onboarding.
- Avoid hype, filler, and vague claims.
- Use headings and lists when they help scanning.
- Keep Markdown prose around 100 columns.
- Use fenced code blocks with a language when one applies.

When a topic is uncertain, document the uncertainty in an issue, ADR, or roadmap item instead of
burying it in user docs.
